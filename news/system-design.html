0. Background

On April 9, 2026, the course released two documents for the System Design phase: the Interface Specification v2.0, which defines how all modules communicate with each other across the system, and the System Design (draft) template, which each team fills in with their own module design.

This document is Team S2's contribution to the Back-end section of the System Design. It specifies the internal architecture of the S2 Data Acquisition module: how sensor data is collected, validated, buffered, and transmitted to downstream components. The design is based on the interface contracts defined in Interface Specification v2.0 and aligns with the functional requirement clustering in the System Design template.

1. S2 in the System Architecture

Team S2 operates in Layer 1 (Sensor) of the three-layer system architecture. Within the Interface Specification v2.0, S2's deliverables appear under "Server calls Database > Sensor Interface", where S2 provides the collect.py program that bridges physical sensors and the server module.

S2's output feeds directly into the Server module (maintained by V1 and V2), which calls collect.py on the embedded platform to receive live sensor samples. This is the core of the IF1 interface contract.

3. Module Specifications

S2-01
Simulated Data Generation

Generates synthetic IMU data that mimics real sensor output, allowing downstream modules (V1, V2, M1, M2) to operate without physical hardware during development and testing. Produces data in the same 45-value CSV format as real sensor packets.

Interface
AttributeSpecification
TriggerSimulated data request from M2 App/frontend
OutputSimulated raw IMU packets to S2-02 (same format as real S1 data)
Format45 comma-separated floating-point values per sample
S2-02
Data Acquisition Core

The central processing module of S2. Receives sensor data from either the real S1 hardware or the simulated data generator (S2-01), validates the data format, and writes validated samples into the asynchronous buffer for transmission. Responds to session start/close commands from M1.

Interface Contract

Data Format: 45 Values per Sample

Each sample contains exactly 45 comma-separated floating-point values representing one time-step of sensor readings from multiple IMU nodes placed on different limb segments. This format is consistent across the entire system: the same 45-value CSV format is used by the predict program's stdin, calibration data files, and training data directories.

# Example output (one sample per line, 45 values each)
0.12,0.34,-9.78,0.01,0.02,-0.03,0.45,...,0.67
0.13,0.35,-9.77,0.02,0.01,-0.02,0.44,...,0.68
# ... continuous streaming
S2-03
Asynchronous Sending

Reads validated samples from the asynchronous buffer and transmits them to V1 AI as formal data. The buffer decouples the data acquisition rate from the network transmission rate, preventing data loss during transient network delays.

Interface
AttributeSpecification
InputValidated samples from asynchronous buffer (FIFO)
ControlStart/Close session request from M1
OutputFormal data to V1 AI (formatted, validated sensor stream)
DeliveryAsynchronous; buffer absorbs rate differences between acquisition and transmission

The asynchronous buffer between S2-02 and S2-03 is an in-memory FIFO queue. This design ensures that sensor data collection is never blocked by network I/O, which is critical for maintaining consistent sampling rates on the embedded platform.
