A. Structural and Specification-Level Issues
OMISSION · A-1

Lack of explicit requirement identifiers and organization

The current SRS presents functionality primarily through use cases, but does not assign unique requirement IDs (e.g., FR-01, FR-02). This makes it difficult to reference, track, and validate requirements across development and testing.

Without identifiers:

Requirements cannot be easily traced to implementation or test cases
Review discussions become ambiguous (“the login requirement” vs a precise ID)

Recommendation:
Introduce a structured requirement list with unique IDs. For example:

FR-01: User registration
FR-02: User login
FR-03: Sensor connection

Additionally, map each requirement to corresponding use cases.

OMISSION · A-2

No explicit system assumptions or operational constraints

The SRS does not document key assumptions about the operating environment.

Missing examples include:

Whether patients always have internet access
Whether sensors are pre-configured or require manual setup
Supported mobile platforms (iOS, Android, versions)

This omission may lead to incorrect design decisions.

Recommendation:
Add a dedicated section:

Assumptions (e.g., “Users have stable internet access during sessions”)
Constraints (e.g., “System must run on Android 10+ and iOS 14+”)
AMBIGUITY · A-3

Unclear system boundary between mobile app and backend services

While components such as M1, M2, S2, V1, and V2 are mentioned, the SRS does not clearly define:

Which functionalities belong to the mobile app vs backend
Where data processing occurs (device vs cloud)

This is particularly problematic in real-time scenarios (e.g., UC-05).

Recommendation:
Clarify:

Data flow boundaries (client-side vs server-side)
Responsibility of each subsystem

Optionally include a high-level architecture diagram.

B. Requirement-Level Issues
AMBIGUITY · B-1

Vague definition of “real-time feedback” in UC-05

UC-05 describes real-time feedback generated by the AI engine, but does not define what “real-time” means.

This creates uncertainty:

Is acceptable latency 100ms? 1s? 5s?
Does feedback need to be continuous or periodic?

Recommendation:
Define measurable constraints, for example:

“Feedback shall be generated within 500ms of receiving sensor data”
“Feedback shall be updated at least 5 times per second during active sessions”
OMISSION · B-2

No data validation rules for sensor input

The SRS assumes sensor data is valid once connected, but does not define:

Acceptable data ranges
Handling of corrupted or missing data
Outlier detection

This is critical because sensor data quality directly affects AI outputs.

Recommendation:
Add validation rules such as:

Range checks (e.g., angle limits)
Missing data handling (interpolation or discard)
Error reporting when data is unreliable
INCONSISTENCY · B-3

Inconsistent level of abstraction in use case descriptions

Some use cases describe high-level user actions (e.g., “log in”), while others include low-level implementation details (e.g., “opens data stream from S2 pipeline”).

This inconsistency leads to:

Uneven readability
Mixing of requirement and design concerns

Recommendation:
Standardize abstraction level:

Keep SRS at requirement level, not implementation level
Move pipeline/component details to design documents
OMISSION · B-4

No definition of failure handling for network disconnection

Several use cases (especially UC-04 and UC-05) assume continuous connectivity, but do not define behavior when:

Bluetooth disconnects
WiFi drops
Server becomes unreachable

This is a critical omission for a real-time rehabilitation system.

Recommendation:
Add alternative flows:

Automatic reconnection attempts
Offline buffering of sensor data
User notification and recovery options
AMBIGUITY · B-5

Unclear role of Medical Professional in real-time interaction

The SRS defines Medical Professional mainly as a reviewer of data, but it is unclear whether they:

Participate in live sessions
Provide real-time feedback
Only interact asynchronously

This ambiguity affects system design and UI requirements.

Recommendation:
Clarify:

Whether Medical Professionals are involved in synchronous sessions
If not, explicitly state that all real-time feedback is AI-driven
OMISSION · B-6

No prioritization of requirements

All use cases are presented as equally important, but in practice:

Some features are core (e.g., sensor connection, session start)
Others are optional (e.g., notifications, advanced analytics)

Without prioritization, development planning is difficult.

Recommendation:
Assign priority levels:

Must-have
Should-have
Nice-to-have
INCONSISTENCY · B-7

Notification behavior is not consistently defined across use cases

UC-07 introduces notifications, but other use cases also imply user feedback (e.g., errors, confirmations) without specifying:

Whether notifications are push-based or in-app
Whether users can configure them

Recommendation:
Define a unified notification model:

Notification types (system, medical, reminder)
Delivery methods (push, in-app)
User preferences
Summary of Findings

The SRS Draft v0.1 demonstrates a solid foundation in identifying core use cases, but still exhibits several issues in:

Requirement precision (lack of measurable definitions)
Completeness (missing assumptions, failure handling, validation rules)
Consistency (mixed abstraction levels and terminology)

These issues fall into the categories of omission, ambiguity, and inconsistency, and addressing them will significantly improve the document’s clarity and engineering value.

A revised version should focus on:

Making requirements testable and measurable
Defining system boundaries and responsibilities clearly
Ensuring consistent abstraction across all sections
