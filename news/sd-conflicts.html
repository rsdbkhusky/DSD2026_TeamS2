1. Overview

During the System Design integration phase, S2 is responsible for consolidating System Design documents from all teams. The primary goal is to determine each team's responsibilities and establish detailed interface specifications so that independently developed components can integrate successfully.

When two teams have each designed their side of a shared interface independently, conflicts are inevitable. This document records all identified conflicts, classifies them, and tracks their resolution.

Conflicts between the designs of two teams are classified into three types:

TypeDefinition
Type A Minor conflict: easy to unify.
Type B Moderate conflict: one team's design involves richer operations or data than the other's.
Type C Critical conflict: require negotiation to reach consensus.

All conflicts are within the IF-S2-V2 interface, covering endpoint paths, HTTP methods, request/response structure, data format, naming conventions, and ID schemes.

2.4 Conflict 3: Session Creation Request Body

Description

S2 sends:

{
  "sessionId": "session-001",      // string, client-provided
  "patientId": "patient-001",      // string
  "sensorJointMapping": {...},     // dict
  "payloadStatus": "bend_knee_10", // string
  "startTime": 1714456800000       // Unix ms
}

V2 expects:

{
  "userId": 1      // integer, references users table
}

Specific conflict points:

  • S2's sessionId is a client-provided string; V2's session id is a server-generated auto-incrementing integer.
  • S2 uses patientId (string); V2 uses userId (integer).
  • S2 sends sensorJointMapping; V2 has no such field.
  • S2 sends payloadStatus; V2 has no such field.
  • S2 sends startTime; V2 auto-generates started_at and does not accept a client-provided value.
  • S2 expects 200 OK or 409 Conflict; V2 returns 201 Created.
Classification Type B S2's design provides richer information, some of which may be utilised by V1. The additional fields do not affect compatibility with V2's current design.
Status Resolved
Resolution Use V2's design

2.5 Conflict 4: Data Upload Structure

Description

S2 sends (three asynchronous lists in one payload):

{
  "sensorData": [
    { "timestamp": ..., "sensorId": "...",
      "accX": ..., "accY": ..., "accZ": ...,
      "gyroX": ..., "gyroY": ..., "gyroZ": ...,
      "roll": ..., "pitch": ..., "yaw": ... }
  ],
  "targetAngles": [
    { "timestamp": ..., "angleID": "...", "angle": ... }
  ],
  "errors": [
    { "timestamp": ..., "sensorId": "...", "errorType": "...", "message": "..." }
  ]
}

V2 expects (single or batch, much simpler):

// single
{ "sessionId": 1, "jointAngles": { "knee": 45.2, "hip": 30.1 }, "isCorrect": true }
// batch
{ "sessionId": 1, "measurements": [
    { "jointAngles": { "knee": 45.2 }, "isCorrect": true }
]}

Specific conflict points:

  • Data granularity is completely different: S2 sends full IMU raw data (accelerometer, gyroscope, orientation angles, independently per sensor); V2 only accepts jointAngles (a free-form JSON object like {"knee": 45.2}). V2 has no storage capability for raw IMU data (no accX/Y/Z, gyroX/Y/Z fields).
  • V2 has no sensorId / deviceId concept; it does not distinguish which sensor produced a reading. V2 also has no storage for sensorJointMapping or payloadStatus.
  • S2's targetAngles list: V2 has no such concept.
  • S2's errors list: V2 has no such concept.
  • V2 has an isCorrect field on each measurement; S2 does not determine movement correctness (this is V1/AI's responsibility).
  • S2's batch upload is one request containing three kinds of asynchronous data; V2's batch upload is an array of the same measurement type.
  • S2 puts sessionId in the URL path; V2 puts sessionId in the request body.
Classification Type C This is the most critical conflict. Upstream, it involves how S1 and S2 actually sample and process sensor data. Downstream, it involves what information V1 needs for AI computation and what M2 needs for display. This requires thorough discussion and negotiation to resolve.
Status Resolved
Resolution Negotiated new format based on S2's design (see implementation guide for details)

2.6 Conflict 5: Session End Request Body

Description

S2 sends:

{ "endTime": 1714456860000, "sampleCount": 1200, "errorCount": 3 }

V2 expects: No request body (just PATCH /sessions/:id/end; the server automatically sets ended_at).

  • V2 does not accept endTime, sampleCount, or errorCount.
  • V2 has no fields for storing sampleCount / errorCount.
Classification Type B It may not be necessary to send sampleCount and errorCount, as these are either unnecessary or can be easily computed on the V2 side. However, whether having the client record session start and end times is a more reasonable design may need to be determined through negotiation.
Status Resolved
Resolution Use V2's design

4.6 Conflict 5: IF-S1-M1 Direct Communication

Description

M1's IS document defines an interface IF-S1-M1 (S1 directly to M1 via BLE), meaning M1 directly receives raw IMU sensor data from S1.

According to the DFD, the data flow should be S1 → S2 → M1. M1 should not communicate directly with S1; instead, S2 acts as the intermediary that processes raw data from S1 and delivers formatted data to M1.

Classification Type C
Status Resolved
Resolution After discussion with M1 lead Yiding WANG: M1 does not communicate directly with S1. All sensor data goes through S2. M1 should remove IF-S1-M1 from their IS document.

5. IF-V2-SelfCorrection

5.1 Overview

V2 provided an Interface Specification document describing their API endpoints. An independent test was conducted against the live V2 backend to verify the actual behavior of all endpoints. This section documents discrepancies between V2's documented interface design and the actual test results, serving as corrections to V2's interface specification.

Tested by Yanbo QIAO and Zhiqi ZHANG.

5.2 Bug 1: Timestamp Format Inconsistency

Description

V2's documentation states "All dates are ISO 8601". However, the actual API responses use two different timestamp formats:

  • timestamp and generated_at fields: standard ISO 8601 (e.g. "2026-05-02T13:43:38.549Z")
  • created_at, started_at, ended_at fields: non-standard format without T separator and Z suffix (e.g. "2026-05-02 13:43:28")

The two formats can be freely converted to each other, so this does not cause functional issues.

Classification V2 implementation bug
Status Identified
Resolution V2 should fix the bug: all timestamp fields should use standard ISO 8601 format. [TODO]
Note: since the two formats can be freely converted, V2's non-standard format ("YYYY-MM-DD HH:MM:SS") may be accepted as a compatible alternative format.

5.3 Conflict 1: Authentication Endpoints Not Documented

Description V2's IS document only lists POST /users (legacy, no password) for user creation. However, V2 has actually implemented a full JWT authentication system with POST /auth/register, POST /auth/login, and GET /auth/me. The IS document is outdated.
Classification Type B V2's actual implementation provides richer functionality than documented.
Status Resolved
Resolution Use V2's actual implementation (JWT authentication endpoints).

5.4 Conflict 2: Legacy POST /users Returns Undocumented Field

Description V2's documentation says POST /users request body is { name, email, role }. The actual response includes an undocumented password: null field.
Classification Type A Minor documentation gap. The extra field does not affect functionality.
Status Resolved
Resolution Use V2's actual implementation. The password: null field can be ignored by consumers.

5.5 Bug 2: userId Type Inconsistency in Engine Endpoint

Description GET /recommendations/engine/:userId returns userId as a string (e.g. "3"), while all other endpoints return user IDs as integers. This is likely caused by the URL path parameter not being converted from string to integer.
Classification V2 implementation bug
Status Identified
Resolution V2 should fix the bug: userId should be int, consistent with all other endpoints. [TODO]

5.6 Bug 3: Field Naming Inconsistency in Push Register

Description POST /push/register response uses userId (camelCase), while all other V2 response fields use snake_case (e.g. user_id).
Classification V2 implementation bug
Status Identified
Resolution V2 should fix the bug: response field should use user_id (snake_case), consistent with all other endpoints. [TODO]

6.2 Conflict 1: User Registration Endpoint

Description
  • M1 IS: uses POST /users with { name, email, role } (no password) to create a user.
  • V2 actual: POST /users is a legacy endpoint. V2 has implemented POST /auth/register with { name, email, password, role }, which returns a JWT token and user object.
Classification Type B V2's actual implementation provides a richer authentication system than what M1's IS document covers.
Status Resolved
Resolution Use V2's design (POST /auth/register with password and JWT). M1 should update their IS document.

6.3 Conflict 2: Missing V2 Endpoints in M1 IS

Description

M1's IS document does not cover the following V2 endpoints that have been verified as operational:

  • Authentication: POST /auth/login, GET /auth/me
  • Session deletion: DELETE /sessions/:id
  • Schedule management: POST /schedule, GET /schedule/:userId, PATCH /schedule/:id, DELETE /schedule/:id
  • Push notifications: POST /push/register, GET /push/tokens/:userId
Classification Type B V2's actual implementation provides more endpoints than M1's IS document covers.
Status Resolved
Resolution Use V2's design. M1 should add these endpoints to their IS document as needed.

8.2 Conflict 1: patientId vs userId in POST /sessions

Description
  • M2: uses patientId in POST /sessions.
  • V2 actual: uses userId.
Classification Type A
Status Resolved
Resolution Use V2's design (userId). M2 should update their document.

8.3 Conflict 2: POST /auth/register Requires File Upload

Description
  • M2: requires POST /auth/register to accept multipart/form-data with a license file field for doctor registration and admin approval.
  • V2 actual: only accepts application/json with { name, email, password, role }. No file upload support.
Classification Type C V2 needs to add file upload support.
Status Pending negotiation
Resolution V2 should add optional license file upload to POST /auth/register. The field should be optional so that patient registration (used by M1, JSON without file) is not affected. [TODO]

8.4 Conflict 3: Doctor Approval Flow Missing in V2

Description

M2 requires a doctor approval workflow:

  • POST /auth/register for doctor should return { userId, status: "pending" } instead of immediately granting a JWT token.
  • POST /auth/login response should include a status field in the user object (e.g. "approved", "pending").
  • GET /auth/status endpoint is needed for checking approval status.

V2 currently has no approval workflow. Registration immediately succeeds and returns a JWT token.
Classification Type C V2 needs to implement an approval flow for doctor accounts.
Status Pending negotiation
Resolution V2 should implement role-based registration: patient registration remains unchanged (immediate success + JWT); doctor registration enters a pending state requiring admin approval. V2 should add status field to user objects and add GET /auth/status endpoint. This does not affect M1 (patient flow unchanged). [TODO]

8.5 Conflict 4: Response Naming Convention

Description M2's document writes GET /measurements/:sessionId response fields in camelCase (e.g. jointAngles, isCorrect). V2's actual responses use snake_case (e.g. joint_angles, is_correct). V2's convention (request camelCase, response snake_case) is consistent across all endpoints and confirmed by all other teams.
Classification Type A
Status Resolved
Resolution Use V2's design (request camelCase, response snake_case). M2 should update their document.

8.6 Conflict 5: POST /recommendations Field Mismatch

Description
  • M2: sends { sessionId, movement, confidence, notes }.
  • V2 actual: accepts { sessionId, movement, confidence, status }.

notes and status are different fields. status is the recommendation state ("pending"/"accepted"/"rejected"); notes is a free-text comment by the doctor. M2 is missing status, and V2 does not have notes.

Classification Type A for missing status (use V2's design). Type C for notes (V2 needs to add).
Status Partially resolved
Resolution M2 should add status field (use V2's design). V2 should add optional notes field to the recommendations table and endpoint. Adding an optional field does not affect other teams. [TODO]

8.7 Conflict 6: GET /recommendations Response Structure

Description
  • M2 expects: { type, content }
  • V2 actual: { id, session_id, movement, status, confidence, created_at }

The response structures are completely different.
Classification Type A
Status Resolved
Resolution Use V2's design. M2 should adapt their frontend to consume V2's response format.

8.8 Conflict 7: Missing V2 Endpoints Required by M2

Description

M2 requires the following endpoints that do not exist in V2:

  • GET /patients: list all patients (filtered by role=patient). M2 also expects an age field in patient data, which V2's users table does not have.
  • GET /patients/:id: get patient details with age field.

Adding age to the users table and adding these endpoints does not affect other teams (M1 simply ignores extra fields).
Classification Type C V2 needs to add these endpoints and the age field.
Status Pending negotiation
Resolution V2 should add GET /patients and GET /patients/:id endpoints (filter users by role=patient), and add age field to the users table. [TODO]

8.9 Conflict 8: Measurements Date Filtering

Description M2 requires the ability to query measurement data filtered by date range (e.g. a specific day or a month). V2's GET /measurements/:sessionId only returns all measurements for a given session, with no date filtering parameters.
Classification Type C V2 needs to add date filtering capability.
Status Pending negotiation
Resolution V2 should add optional date range query parameters (e.g. ?startDate=...&endDate=...) to measurement retrieval endpoints. Adding optional parameters does not affect other teams. [TODO]
