System Design Conflicts and Solutions

Cross-team integration of System Design documents. Identifies interface conflicts between S2 and other teams, classifies each conflict, and tracks resolutions.

DateAuthorsDescription
Apr 24, 2026Zhiqi ZHANGInitial conflict analysis for IF-S2-V2

1. Overview

During the System Design integration phase, S2 is responsible for consolidating System Design documents from all teams. The primary goal is to determine each team's responsibilities and establish detailed interface specifications so that independently developed components can integrate successfully.

When two teams have each designed their side of a shared interface independently, conflicts are inevitable. This document records all identified conflicts, classifies them, and tracks their resolution.

Conflicts between the designs of two teams are classified into three types:

TypeDefinition
Type A One team's design involves richer operations or data than the other's.
Type B The two designs are inconsistent (not simply one being richer than the other), but easy to unify.
Type C The two designs are inconsistent (not simply one being richer than the other), and require negotiation to reach consensus.

2. IF-S2-V2

2.1 Overview

Comparing the S2 Interface Specification (Section 2.4, IF-S2-V2) with V2's Backend Integration Guide, 9 conflicts were identified. All conflicts are within the IF-S2-V2 interface, covering endpoint paths, HTTP methods, request/response structure, data format, naming conventions, and ID schemes.

2.4 Conflict 3: Session Creation Request Body

Description

S2 sends:

{
  "sessionId": "session-001",      // string, client-provided
  "patientId": "patient-001",      // string
  "sensorJointMapping": {...},     // dict
  "payloadStatus": "bend_knee_10", // string
  "startTime": 1714456800000       // Unix ms
}

V2 expects:

{
  "userId": 1      // integer, references users table
}

Specific conflict points:

  • S2's sessionId is a client-provided string; V2's session id is a server-generated auto-incrementing integer.
  • S2 uses patientId (string); V2 uses userId (integer).
  • S2 sends sensorJointMapping; V2 has no such field.
  • S2 sends payloadStatus; V2 has no such field.
  • S2 sends startTime; V2 auto-generates started_at and does not accept a client-provided value.
  • S2 expects 200 OK or 409 Conflict; V2 returns 201 Created.
Classification Type A S2's design provides richer information, some of which may be utilised by V1. The additional fields do not affect compatibility with V2's current design.

2.5 Conflict 4: Data Upload Structure

Description

S2 sends (three asynchronous lists in one payload):

{
  "sensorData": [
    { "timestamp": ..., "sensorId": "...",
      "accX": ..., "accY": ..., "accZ": ...,
      "gyroX": ..., "gyroY": ..., "gyroZ": ...,
      "roll": ..., "pitch": ..., "yaw": ... }
  ],
  "targetAngles": [
    { "timestamp": ..., "angleID": "...", "angle": ... }
  ],
  "errors": [
    { "timestamp": ..., "sensorId": "...", "errorType": "...", "message": "..." }
  ]
}

V2 expects (single or batch, much simpler):

// single
{ "sessionId": 1, "jointAngles": { "knee": 45.2, "hip": 30.1 }, "isCorrect": true }
// batch
{ "sessionId": 1, "measurements": [
    { "jointAngles": { "knee": 45.2 }, "isCorrect": true }
]}

Specific conflict points:

  • Data granularity is completely different: S2 sends full IMU raw data (accelerometer, gyroscope, orientation angles, independently per sensor); V2 only accepts jointAngles (a free-form JSON object like {"knee": 45.2}). V2 has no storage capability for raw IMU data (no accX/Y/Z, gyroX/Y/Z fields).
  • V2 has no sensorId / deviceId concept; it does not distinguish which sensor produced a reading. V2 also has no storage for sensorJointMapping or payloadStatus.
  • S2's targetAngles list: V2 has no such concept.
  • S2's errors list: V2 has no such concept.
  • V2 has an isCorrect field on each measurement; S2 does not determine movement correctness (this is V1/AI's responsibility).
  • S2's batch upload is one request containing three kinds of asynchronous data; V2's batch upload is an array of the same measurement type.
  • S2 puts sessionId in the URL path; V2 puts sessionId in the request body.
Classification Type C This is the most critical conflict. Upstream, it involves how S1 and S2 actually sample and process sensor data. Downstream, it involves what information V1 needs for AI computation and what M2 needs for display. This requires thorough discussion and negotiation to resolve.

2.6 Conflict 5: Session End Request Body

Description

S2 sends:

{ "endTime": 1714456860000, "sampleCount": 1200, "errorCount": 3 }

V2 expects: No request body (just PATCH /sessions/:id/end; the server automatically sets ended_at).

  • V2 does not accept endTime, sampleCount, or errorCount.
  • V2 has no fields for storing sampleCount / errorCount.
Classification Type A It may not be necessary to send sampleCount and errorCount, as these are either unnecessary or can be easily computed on the V2 side. However, whether having the client record session start and end times is a more reasonable design may need to be determined through negotiation.

2.7 Conflict 6: Timestamp Format

Description
  • S2: Unix millisecond timestamps (integer), e.g. 1714456800000
  • V2: ISO 8601 strings (UTC), e.g. "2025-04-21T14:30:00.000Z"

This discrepancy applies to all time-related fields across all endpoints.
Classification Type B V2's design (ISO 8601) is more reasonable for a REST API.
