Software Requirements Specification — Draft

The first draft of the SRS for the Limb Motion Recognition & Assistant system. This document identifies the key actors, use cases, and interaction flows across all three system layers — Sensor, Server, and Monitor.

DateAuthorDescription
Apr 5, 2026Zhiqi ZHANG (S2 PM)Initial draft — background, actors, use cases UC-01 through UC-07

0. Background

The Limb Motion Recognition & Assistant system is a distributed rehabilitation platform jointly developed by UTAD (Portugal) and Jilin University (China) under the DSD 2025–2026 programme. The system captures joint motion data from wearable IMU sensors worn by rehabilitation patients, processes that data through AI-based motion recognition on a central server, and delivers real-time feedback to patients and historical analysis to medical professionals.

This Software Requirements Specification (SRS) defines the functional requirements of the complete system through a set of use cases. It is intended to serve as the shared contract between all six development teams (S1, S2, V1, V2, M1, M2) — ensuring that each team builds towards the same user-facing goals, even when working independently on different layers.

The system must support two deployment contexts that impose different requirements on connectivity, guidance, and data handling:

  • Clinical setting (hospital/rehabilitation centre): Medical staff assist with sensor setup. Stable network environment. Sessions are supervised. Real-time data upload is assumed.
  • Home setting: Patients operate independently. Network may be unstable (home WiFi or mobile data). The system must guide users through setup and tolerate intermittent connectivity with local data buffering.

1. Actors

Three primary actors interact with the system:

Patient
A rehabilitation patient wearing IMU sensors. Uses the mobile app (M1) to perform guided exercises and receive real-time posture feedback. May operate at home or in a clinical setting.
Medical Professional
A doctor or physiotherapist who reviews recovery data, prescribes exercise plans, and monitors patient progress through the clinical web dashboard (M2).
System
The automated backend — sensor data pipeline (S2), AI inference engine (V1), API gateway (V2) — that processes data and drives the user-facing interfaces.

3. Use Case Specifications

UC-01
User Registration
Version 0.1
ActorsPatient, Medical Professional
GoalAllow a new user to create an account and join the system
TriggerUser opens the app or dashboard for the first time and selects "Register"
PreconditionThe system is online and the registration service is available
PostconditionA unique user account is created and stored; user is redirected to login
FrequencyOnce per user, at initial setup
Basic Flow
ActorSystem
User selects role (Patient or Medical Professional) and enters name, email, and password
Validates that the email address is not already registered
Validates password meets complexity requirements (min 8 characters, at least one number)
Creates user account and persists to database; assigns role-based permissions
Sends confirmation email to the registered address
User receives confirmation and is directed to the login screen
Alternative Flow — Email Already Registered
ActorSystem
User submits registration form
Detects that the email is already associated with an existing account
Returns error: "An account with this email already exists. Please log in or reset your password."
User is offered links to Login or Forgot Password
Alternative Flow — Invalid Password Format
ActorSystem
User submits a password that does not meet complexity requirements
Returns inline validation error highlighting the password field with specific requirement not met
User corrects the password and resubmits
UC-02
User Login
Version 0.1
ActorsPatient, Medical Professional
GoalAuthenticate an existing user and grant access to role-appropriate features
TriggerUser opens the app or dashboard and selects "Log In"
PreconditionUser has a registered account; system is online
PostconditionUser is authenticated and redirected to their role-specific home screen
FrequencyEvery session start
Basic Flow
ActorSystem
User enters registered email and password
Verifies credentials against stored records
Issues authentication token; records login timestamp
Redirects Patient to M1 home screen; Medical Professional to M2 dashboard
Alternative Flow — Incorrect Credentials
ActorSystem
User submits incorrect email or password
Returns generic error: "Incorrect email or password" (without specifying which field is wrong, for security)
After 5 consecutive failed attempts, locks the account for 15 minutes and notifies the user
UC-03
Sensor Connection
Version 0.1
ActorsPatient (with or without clinical staff assistance)
GoalEstablish a verified connection between the IMU sensor(s) and the system before a session begins
TriggerPatient selects "Start Session" in the mobile app
PreconditionPatient is logged in; IMU sensor is powered on and within Bluetooth range
PostconditionSensor is paired, calibrated, and streaming valid data; session can begin
FrequencyOnce per rehabilitation session
Basic Flow — Clinical Setting (staff-assisted)
ActorSystem
Clinical staff attaches IMU sensor to patient's limb and powers it on
Patient (or staff) taps "Connect Sensor" in the app
Scans for available Bluetooth devices and identifies the sensor by device ID
Establishes Bluetooth connection and begins receiving raw IMU data
Runs calibration routine: prompts patient to hold limb still for 3 seconds to establish baseline orientation
Patient holds still while the calibration countdown runs
Confirms calibration success; displays sensor status as "Connected & Ready"
Basic Flow — Home Setting (patient self-setup)
ActorSystem
Patient opens app; app detects no sensor is connected
Displays illustrated step-by-step guide for attaching the sensor to the correct limb position
Patient attaches sensor following the guide and powers it on
Patient taps "Connect"
Scans, pairs, calibrates (same as clinical flow)
Confirms connection; if network is unavailable, enables local buffering mode and notifies patient that data will sync when connectivity is restored
Alternative Flow — Sensor Not Found
ActorSystem
Patient taps "Connect Sensor"
Scans for 10 seconds and finds no compatible device
Displays troubleshooting guide: check power, check Bluetooth is enabled, bring sensor within 1 metre
Patient follows guide and retries
UC-04
Start Rehabilitation Session
Version 0.1
ActorsPatient, System
GoalBegin a timed rehabilitation exercise session with live sensor data flowing to the server
TriggerSensor is connected (UC-03 complete); patient taps "Begin Exercise"
PreconditionPatient is logged in, sensor is connected and calibrated, an exercise plan has been prescribed (UC-07) or a default plan exists
PostconditionSession is active; sensor data is streaming through S2 pipeline to V1 for inference; M1 is displaying real-time feedback
FrequencyMultiple times per day per patient
Basic Flow
ActorSystem
Patient selects the prescribed exercise from their plan and taps "Begin"
Displays a 3-second countdown and prompts patient to assume starting position
Patient assumes starting position
Begins recording session; opens data stream from S2 pipeline to V1; creates session record in V2 database with timestamp and patient ID
Patient performs the exercise
Continuously forwards sensor packets through S2 → V1 → V2 → M1 in real time
Patient completes the exercise set and taps "End Session"
Closes the data stream; finalises session record; displays session summary (duration, rep count, average accuracy)
Alternative Flow — Connection Lost Mid-Session (Home Setting)
ActorSystem
Patient is mid-session when network connectivity drops
Switches to local buffering mode silently; displays a small offline indicator but does not interrupt the session
Patient completes the session normally
When connectivity is restored, uploads buffered session data to V2 in the background
UC-05
Receive Real-Time Motion Feedback
Version 0.1
ActorsPatient, System (V1 AI engine, M1 mobile app)
GoalProvide the patient with immediate, actionable feedback on joint angle accuracy during exercise
TriggerActive session (UC-04); V1 inference result is returned for each sensor packet
PreconditionSession is active; data pipeline S2 → V1 → V2 → M1 is operational
PostconditionPatient has received visual and/or haptic feedback indicating correctness of movement
FrequencyContinuous during active session (driven by sensor sampling rate)
Basic Flow
ActorSystem
Patient performs a repetition of the prescribed movement
S2 acquires sensor packet and forwards it through the pipeline
V1 runs inference; compares detected joint angles against prescribed target angles for this exercise
V1 produces a result: angle deviation per joint, overall accuracy score, and any posture alert flags
M1 renders the result on screen: colour-coded joint overlay (green = within tolerance, red = deviation), text guidance (e.g. "Raise your knee 10° higher"), and optional haptic pulse
Patient reads feedback and adjusts their movement for the next repetition
Alternative Flow — Posture Alert (dangerous deviation)
ActorSystem
Patient performs a movement with joint angle deviation exceeding the safety threshold
V1 flags a posture alert; M1 interrupts the exercise with a full-screen warning, strong haptic feedback, and audio alert
Session is paused automatically; patient is instructed to rest and return to starting position
Patient acknowledges the alert
Alert event is logged to V2 and flagged for review by the assigned Medical Professional
UC-06
Review Patient Progress
Version 0.1
ActorsMedical Professional
GoalReview a patient's rehabilitation history, session accuracy trends, and flagged alerts via the clinical dashboard
TriggerMedical Professional selects a patient from their patient list in M2
PreconditionMedical Professional is logged in; patient has at least one completed session
PostconditionMedical Professional has viewed the patient's progress data and optionally added clinical notes
FrequencyAt each clinical review appointment; may also be triggered by a posture alert notification
Basic Flow
ActorSystem
Medical Professional logs into M2 dashboard and selects a patient
Fetches patient's session history from V2 REST API; renders recovery trend charts (joint angle accuracy over time, session frequency, alert history)
Medical Professional reviews the charts and drills into a specific session
Displays detailed session view: per-repetition accuracy, 3D limb reconstruction replay (M2), timeline of any alerts triggered during the session
Medical Professional adds a clinical note to the session record
Saves the note to V2 and associates it with the patient record
Alternative Flow — Alert Notification Triggered Review
ActorSystem
Medical Professional receives a push notification: "Patient [Name] triggered a posture alert during today's session"
Medical Professional taps the notification
Opens M2 directly to the flagged session with the alert moment highlighted in the timeline
Medical Professional reviews the event and contacts patient if necessary
UC-07
Prescribe Exercise Plan
Version 0.1
ActorsMedical Professional
GoalDefine and assign a personalised exercise plan to a patient, including exercise types, target angles, repetition counts, and schedule
TriggerMedical Professional selects "New Plan" from a patient's profile in M2
PreconditionMedical Professional is logged in; patient account exists in the system
PostconditionExercise plan is saved and immediately visible to the patient in M1; future sessions will reference this plan for target angles and guidance
FrequencyAt initial assessment and whenever the patient's recovery stage changes
Basic Flow
ActorSystem
Medical Professional opens patient profile and selects "Create Exercise Plan"
Presents plan builder: list of available exercises, target joint, target angle range, repetitions per set, sets per session, weekly frequency
Medical Professional configures the plan parameters for each prescribed exercise
Medical Professional saves and publishes the plan
Stores the plan in V2; pushes a notification to the patient's M1 app: "Your physiotherapist has updated your exercise plan"
V1 inference engine loads the new target angle parameters for this patient's future sessions
Alternative Flow — Plan Conflicts with Patient's Current Capability
ActorSystem
Medical Professional sets a target angle range that exceeds the patient's recorded maximum range of motion from previous sessions
Displays a warning: "Target angle (95°) exceeds patient's recorded maximum (72°) from the last 7 days. Confirm or adjust."
Medical Professional reviews the warning and either adjusts the target or confirms the override with a clinical justification note
